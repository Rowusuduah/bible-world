# BibleWorld Micro-Core and Deep Research Protocol

Purpose: keep BibleWorld focused on structurally grounded insight, real technical gaps, and buildable outcomes.

Micro-core rules:
1. One cycle, one structural thesis.
2. Start with the smallest correct artifact first: pattern note, teardown, or build spec.
3. Do not force mappings. If scripture and the technical gap do not share structure, say so.
4. Separate the structural insight from product polish. If the structural insight is weak, stop there.
5. Every build candidate must answer: what is the documented gap, what is the structural scripture pattern, and why this is different from current tools.

Deep research rules:
1. For any Level 3 pattern or build candidate, use at least 3 fresh sources when possible.
2. Include at least 1 primary source from company docs, research papers, maintainer docs, or a first-party announcement whenever available.
3. Record source date, source type, competitor notes, contradiction notes, and confidence level.
4. When current tools already solve the problem well, say so and downgrade the claim.
5. If no structural match exists, write NO STRUCTURAL MATCH instead of stretching the reasoning.

Required cycle report blocks:
- Core Thesis
- Research Ledger
- Benchmark Check
- Reproducibility Block

Research Ledger minimum fields:
- Gap tested
- Structural match tested
- Sources used
- Freshest source date
- Competitors checked
- Contradictions found
- Confidence level

Reproducibility Block minimum fields:
- Cycle ID
- Prompt version
- Freshest source date
- Benchmark items run
- Files updated
